## Slide 1 – Why Integrate R into a SAS Workflow?

The pharma industry runs on SAS for regulatory submissions (CDISC, eCTD).
R has become the go-to for:
- Specialized statistical methods (MMRM via `mmrm`, `nlme`)
- Cutting-edge survival models (`flexsurv`, `survminer`)
- Publication-quality graphics (`ggplot2`, `forestplot`)
- Pharmaverse packages (`admiral`, `rtables`, `tern`, `teal`)

**The reality:** Many teams use both. The question is how to make them talk to each other
without leaving the SAS session.

Key PharmaSUG themes driving this:
- Dual-language validation (SAS primary, R independent QC)
- Regulatory acceptance of R output growing (FDA pilot submissions in R)
- Biostatisticians prefer R; programmers prefer SAS – one workflow needed

## Slide 3 – Three Ways to Call R from SAS

| Feature | PROC IML SUBMIT/R | R Runner (Custom Step) | PROC R (NEW – Viya 2026.03) |
|---------|-------------------|------------------------|------------------------------|
| Available since | SAS 9.2 (2008) | Viya 2023.08 (community) | Viya Stable 2026.03 |
| Interface | Matrix language wrapper | Low-code drag-and-drop | Native SAS procedure |
| Data transfer | `ExportMatrixToR` / `ImportMatrixFromR` | SAS dataset → R data frame (via rpy2/Python) | `sd2df()` / `df2sd()` callbacks |
| Macro variable exchange | Via IML scalar → R scalar | Limited | `symget()` / `symput()` / `submit()` callbacks |
| Requires Python? | No | Yes (rpy2) | No |
| Best for | Statisticians already in IML | Flow-based pipelines, non-coders | General clinical programmers |
| Admin setup | R_HOME + RLANG option | sas-pyconfig + rpy2 | Enable PROC R + sas-pyconfig |
| Code style | Embedded in PROC IML block | GUI / .sas flow | Familiar PROC syntax |

## Slide 4 – PROC IML SUBMIT/R: How It Works

The oldest and most battle-tested approach. Requires `RLANG` system option.

```sas
options rlang;   /* must be set at SAS startup or in sasv9.cfg */

proc iml;
  /* 1. Read SAS data into IML matrices */
  use adlb where (paramcd = "HBA1C" and ablfl ne "Y");
  read all var {usubjid avisit chg base trt01p} into X[colname=varnames];
  close adlb;

  chg  = X[, loc(varnames = "chg")];
  base = X[, loc(varnames = "base")];

  /* 2. Push matrices to R */
  call ExportMatrixToR(chg,  "chg");
  call ExportMatrixToR(base, "base");

  /* 3. Run R code */
  submit / R;
    library(ggplot2)
    df <- data.frame(base = base, chg = chg)
    p  <- ggplot(df, aes(x = base, y = chg)) +
            geom_point(alpha = 0.5) +
            geom_smooth(method = "lm") +
            labs(title = "Baseline vs Change in HbA1c")
    ggsave("output/baseline_vs_change.png", p, width=6, height=4)
  endsubmit;

  /* 4. Pull results back */
  submit / R;
    fit    <- lm(chg ~ base)
    coefs  <- as.numeric(coef(fit))
  endsubmit;
  call ImportMatrixFromR(coefs, "coefs");
  print coefs[c={"Intercept" "Slope"}];
quit;
```

Limitations:
- Only matrices/vectors transfer (no data frames directly)
- R graphics saved to file, not embedded in ODS
- Requires `RLANG` option enabled at server level

## Slide 5 – R Runner Custom Step: How It Works

R Runner (github.com/SundareshSankaran/r-runner) is a SAS Studio Custom Step
that wraps R execution via the `rpy2` Python bridge.

Architecture:
```
SAS Studio Flow
    │
    ├─ Input Port  (SAS dataset)  ──► converted to R data frame
    │
    ├─ R Snippet / R Script file
    │       └─ R code runs in rpy2 session
    │
    └─ Output Port (SAS dataset) ◄── R data frame written back
```

Equivalent SAS code generated by the step:
```sas
/* R Runner generates PROC PYTHON internally */
proc python;
submit;
import rpy2.robjects as ro
from rpy2.robjects import pandas2ri
import pandas as pd

pandas2ri.activate()

# Input SAS dataset already converted to pandas df: r_input_table
r_code = """
  library(ggplot2)
  p <- ggplot(r_input_table, aes(x=BASE, y=CHG, color=TRT01P)) +
         geom_boxplot() +
         labs(title="HbA1c Change by Visit and Treatment")
  ggsave('/output/boxplot.png', p)
"""
ro.r(r_code)
endsubmit;
run;
```

Key points:
- Requires Python + rpy2 installed via sas-pyconfig
- Great for non-programmers using SAS Studio Flows
- Data transfer goes SAS → Python (pandas) → R (data frame)
- Output R data frame → pandas → SAS dataset

## Slide 6 – PROC R: The New Native Procedure (Viya 2026.03)

PROC R is a first-class SAS procedure introduced in SAS Viya Stable 2026.03.
It provides the cleanest integration: no IML wrapper, no Python bridge.

R code goes between `SUBMIT` and `ENDSUBMIT`. PROC R exposes callback methods
for all data and macro variable exchange.

### Statement options

```sas
proc r;            /* start / reuse existing R subprocess (shared session) */
proc r restart;    /* fresh R session – clears all R objects               */
proc r terminate;  /* shut down R subprocess, release resources            */
proc r inFile=ref; /* execute an external .r script file                   */
```

Multiple `proc r` calls in the same SAS session share the same R subprocess.
Use `restart` for a clean QC run; use `terminate` when fully done.

### Basic syntax

```sas
proc r;
  submit;
    print("Hello from R — inside SAS!")
  endsubmit;
run;
/* Log: [1] "Hello from R — inside SAS!"  */
/* Log: NOTE: R initialized.              */
```

### Data transfer: sd2df() and df2sd()

```sas
proc r;
  submit;
    # sd2df: dataset name, then libname as separate arguments
    df <- sd2df("adlb", "work")
    cat("Rows:", nrow(df), "\n")

    # process...
    out <- aggregate(CHG ~ TRT01P + AVISIT, df, mean)

    # df2sd: data frame, dataset name, libname
    df2sd(out, "work.r_means")
  endsubmit;
run;
```

### Macro variable exchange

```sas
%let alpha = 0.05;

proc r;
  submit;
    alpha_val <- as.numeric(symget("alpha"))   # read from SAS
    df        <- sd2df("adlb", "work")
    t_res     <- t.test(CHG ~ TRT01P, data = df)
    pval      <- round(t_res$p.value, 4)
    symput("r_pvalue", as.character(pval))     # write back to SAS
    submit(paste0("%let r_n=", nrow(df), ";")) # execute SAS code from R
  endsubmit;
run;
%put NOTE: p = &r_pvalue.  N = &r_n.;
```

### Running an external R script

```sas
filename rscript '/path/to/analysis.r';
/* or from SAS Content: */
filename rscript filesrvc folderPath='/Users/me/My Folder' filename='analysis.r';

proc r inFile=rscript;
  submit;
  endsubmit;
run;
```

### Checking the R environment

```sas
%let rpath = %sysget(PROC_RPATH);
%put NOTE: PROC_RPATH = &rpath.;
```

## Slide 7 – Passing Data Between SAS and R: Summary

### SAS → R

| Method | How |
|--------|-----|
| PROC IML | `call ExportMatrixToR(matrix, "r_name")` |
| R Runner | Input port (SAS dataset auto-converted via rpy2) |
| PROC R | `sd2df("dataset", "libname")` — then `names(df) <- toupper(names(df))` |

### R → SAS

| Method | How |
|--------|-----|
| PROC IML | `call ImportMatrixFromR(iml_matrix, "r_name")` |
| R Runner | Output port; specify R data frame name in step parameter |
| PROC R | `df2sd(df, "dataset", "libname")` inside SUBMIT block |

### Macro Variables

| Method | How |
|--------|-----|
| PROC IML | Export scalar via `ExportMatrixToR`; import back then `call symputx` |
| R Runner | Limited; environment variable workarounds |
| PROC R | `symget("macvar")` reads; `symput("macvar","val")` writes; `submit("%let x=v;")` executes SAS |

### Calling SAS Functions from R (PROC R only)

```r
sasfnc("exist", "work", "LIBREF")   /* check if libref exists */
sasfnc("today")                      /* call SAS TODAY() function */
```

### Rendering Output (PROC R only)

```r
rplot(ggplot_object)                            /* render in Results window  */
rplot(ggplot_object, filename = "plot.png")     /* render + save a copy      */
renderImage(paste0(sas$workpath, "plot.png"))   /* render a saved image file */
show(df, title = "My Table", count = 10)        /* display R object          */
```

### Checking the R environment

```sas
%let rpath = %sysget(PROC_RPATH);
%put NOTE: PROC_RPATH = &rpath.;
```

## Slide 8 – Case Study: MMRM Model Comparison

Scenario: Compare MMRM estimates from SAS PROC MIXED vs R `mmrm` package.

### SAS PROC MIXED (reference model)
```sas
proc mixed data=adlb_mmrm;
  class usubjid trt01p avisit(ref="BASELINE");
  model chg = base trt01p avisit trt01p*avisit / ddfm=kr;
  repeated avisit / subject=usubjid type=un;
  lsmeans trt01p*avisit / diff cl alpha=0.05;
  ods output lsmeans=work.sas_lsmeans diffs=work.sas_diffs;
run;
```

### R mmrm (independent QC)
```sas
proc r;
  submit;
    library(mmrm)
    df <- sd2df("work.adlb_mmrm")
    df$AVISIT  <- factor(df$AVISIT, levels=c("BASELINE","WEEK08","WEEK16","WEEK24"))
    df$TRT01P  <- factor(df$TRT01P, levels=c("DRUG A","PLACEBO"))
    df$USUBJID <- factor(df$USUBJID)

    fit <- mmrm(
      formula = CHG ~ BASE + TRT01P + AVISIT + TRT01P:AVISIT +
                us(AVISIT | USUBJID),
      data    = df,
      method  = "Kenward-Roger"
    )
    est        <- as.data.frame(summary(fit)$coefficients)
    est$param  <- rownames(est)
    df2sd(est, "work.r_mmrm_est")
  endsubmit;
run;
```

### Comparison step
```sas
proc sql;
  create table work.model_compare as
  select
    coalesce(s.effect, r.param) as param,
    s.estimate   as sas_est,
    r.Estimate   as r_est,
    abs(s.estimate - r.Estimate) as diff,
    case when abs(s.estimate - r.Estimate) > 0.001
         then "REVIEW" else "OK" end as status
  from work.sas_lsmeans s
  full join work.r_mmrm_est r on s.effect = r.param;
quit;

proc print data=work.model_compare; run;
```

## Slide 9 – Kaplan-Meier with R survminer

```sas
proc r;
  submit;
    library(survival)
    library(survminer)

    df      <- sd2df("work.adtte")
    df      <- df[df$PARAMCD == "OS", ]
    df$TRT01P <- factor(df$TRT01P, levels = c("DRUG A","PLACEBO"))

    fit <- survfit(Surv(AVAL, CNSR==0) ~ TRT01P, data = df)

    # KM plot rendered directly in SAS Results window
    p <- ggsurvplot(fit, data=df, pval=TRUE, conf.int=TRUE,
                    risk.table=TRUE,
                    title="Overall Survival by Treatment")
    rplot(p)

    # Log-rank p-value back to SAS
    lr   <- survdiff(Surv(AVAL, CNSR==0) ~ TRT01P, data=df)
    pval <- round(1 - pchisq(lr$chisq, df=1), 4)
    symput("r_logrank_p", as.character(pval))

    # Median survival table → SAS dataset
    med        <- as.data.frame(surv_median(fit))
    med$STRATA <- gsub("TRT01P=","", rownames(med))
    df2sd(med, "work.r_km_median")
  endsubmit;
run;

%put NOTE: Log-rank p = &r_logrank_p.;

proc print data=work.r_km_median label;
  title "Median Overall Survival (R survminer)";
run;
```

## Slide 10 – Storing R Results Back as SAS Tables

Best practice: always write R results to a permanent SAS library.

```sas
libname results "/clinical/study001/results";

proc r;
  submit;
    library(mmrm)
    library(broom)

    df <- sd2df("work.adlb_mmrm")
    df$AVISIT  <- factor(df$AVISIT)
    df$TRT01P  <- factor(df$TRT01P)
    df$USUBJID <- factor(df$USUBJID)

    fit <- mmrm(CHG ~ BASE + TRT01P + AVISIT + TRT01P:AVISIT +
                us(AVISIT | USUBJID), data=df, method="Kenward-Roger")

    out <- as.data.frame(tidy(fit, conf.int = TRUE))

    # Write directly to permanent library
    df2sd(out, "results.mmrm_estimates")
    cat("Written to results.mmrm_estimates\n")
  endsubmit;
run;

proc contents data=results.mmrm_estimates; run;
proc print data=results.mmrm_estimates(obs=10); run;
```

## Slide 11 – PROC R Callback Methods: Quick Reference

All callbacks are available inside the SUBMIT block. They are case-sensitive.

| Callback | Purpose | Example |
|----------|---------|---------|
| `sd2df("ds", "lib")` | SAS dataset → R data frame | `df <- sd2df("adlb", "work")` |
| `df2sd(df, "ds", "lib")` | R data frame → SAS dataset | `df2sd(out, "results", "work")` |
| `symget("macvar")` | Read SAS macro variable | `a <- as.numeric(symget("alpha"))` |
| `symput("macvar","val")` | Write R value → SAS macro var | `symput("pval", as.character(p))` |
| `submit("sas code")` | Execute SAS code from R | `submit("%let n=100;")` |
| `sasfnc("func"<,args>)` | Call a SAS function | `sasfnc("today")` |
| `rplot(obj<,filename=f>)` | Render R plot in Results | `rplot(p, filename="plot.png")` |
| `renderImage("path")` | Render saved image in Results | `renderImage(paste0(sas$workpath,"p.png"))` |
| `show(obj,title=,count=)` | Display R object in Results | `show(df, title="Cars", count=10)` |

`sas$workpath` — path to the SAS WORK directory.

`proc r restart;` — fresh R session, clears all R objects.

`proc r terminate;` — shuts down R subprocess, releases resources.

`proc r inFile=ref;` — executes an external `.r` script file.

`%sysget(PROC_RPATH)` — SAS macro function to check which R is active.

## Slide 12 – Bonus: Polyglot Pipeline (SAS + Python + R)

All three languages can collaborate in a single SAS session, sharing WORK and macro variables.

```
[SAS]    Clean data, descriptive stats
   ↓  work.adeff_clean
[Python] Feature engineering + sklearn model
         SAS.sd2df() / SAS.df2sd() / SAS.symput()
   ↓  work.adeff_preds  +  &py_r2.  &py_mae.
[R]      ggplot2 visualizations of model results
         sd2df() reads Python output; symget() reads Python metrics
         rplot() renders directly in SAS Results
```

Note the difference in Python vs R callback syntax:

| | Python (PROC PYTHON) | R (PROC R) |
|-|----------------------|------------|
| SAS dataset → language | `SAS.sd2df("work.ds")` | `sd2df("work.ds")` |
| language → SAS dataset | `SAS.df2sd(df, "work.ds")` | `df2sd(df, "work.ds")` |
| Read macro variable | `SAS.symget("macvar")` | `symget("macvar")` |
| Write macro variable | `SAS.symput("macvar","v")` | `symput("macvar","v")` |

See `05_Demo_Polyglot.sas` for the full working example.

## Slide 14 – Key Takeaways

1. PROC IML SUBMIT/R is mature and works well for matrix-level exchange; it is the right choice when you are already inside PROC IML.

2. R Runner fills a gap for SAS Studio Flow users who want a drag-and-drop experience, but it depends on Python/rpy2 and is a community tool.

3. PROC R (Viya 2026.03) is the strategic path forward: native SAS syntax, full dataset transfer via `sd2df`/`df2sd`, macro variable exchange via `symget`/`symput`, no Python dependency.

4. PROC R's `rplot()` and `renderImage()` bring R graphics directly into the SAS Results window. `rplot(p, filename="f.png")` renders and saves in one call.

5. `proc r restart;` gives a clean R session for independent QC runs. `proc r terminate;` releases resources when done.

6. `inFile=` lets you execute existing `.r` scripts — no need to paste R code into SAS programs.

7. SAS, Python, and R can all collaborate in one session. Data and macro variables flow freely between all three languages.

8. Always write R results back to a permanent SAS library with `df2sd()` to maintain a single source of truth for TFL production.
