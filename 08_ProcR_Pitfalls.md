# PROC R – Common Pitfalls

Encountered during live coding with SAS Viya Stable 2026.03.

---

## 1. Column names after sd2df() are not guaranteed lowercase

`sd2df()` returns column names **exactly as stored in the SAS dataset**.
- User `data` step variables → typically lowercase (`trt01p`, `chg`)
- SAS-supplied datasets (`sashelp.*`) → may be mixed case (`Sex`, `Age`, `Weight`)

**Symptom:** `Error: Column 'TRT01P' is not found` or `KeyError: 'weight'`

**Fix – R:** normalise immediately after every `sd2df()` call:
```r
df <- sd2df("adlb", "work")
names(df) <- toupper(names(df))   # then reference as TRT01P, CHG, etc.
```

**Fix – Python (PROC PYTHON):** same rule, lowercase convention:
```python
df = SAS.sd2df("work.adeff_clean")
df.columns = df.columns.str.lower()
```

## 2. df2sd() creates $32767 character columns

`df2sd()` uses a parquet file as an intermediate format. The parquet engine
in SAS does not preserve character column lengths — it always produces `$32767`
regardless of the actual string content. This happens even after `substr()`
truncation in R, because the length metadata is lost in the parquet round-trip.

The fix requires two steps: R generates a SAS `LENGTH` statement string and
writes it to a macro variable via `symput()`; a DATA step then applies it
after `df2sd()` returns.

**Symptom:** `proc contents` shows character variables with length 32767.

**Fix — add `sas_length_stmt()` helper in R:**
```r
sas_length_stmt <- function(df, min_width = 1L) {
  char_cols <- names(df)[sapply(df, is.character)]
  if (length(char_cols) == 0) return("")
  parts <- sapply(char_cols, function(col) {
    max_len <- max(nchar(df[[col]], type = "bytes"), na.rm = TRUE)
    max_len <- max(max_len, min_width, na.rm = TRUE)
    paste0(col, " $", max_len)
  })
  paste("length", paste(parts, collapse = " "), ";")
}
```

**Pattern — R side:**
```r
df <- trim_char_widths(df)
symput("my_lengths", sas_length_stmt(df))
df2sd(df, "my_ds_raw", "work")
```

**Pattern — SAS side (immediately after the proc r block):**
```sas
data work.my_ds;
  &my_lengths.          /* LENGTH statement generated by R */
  set work.my_ds_raw;
run;
```

The `LENGTH` statement must appear before the `SET` statement so SAS
allocates the correct buffer width before reading the data.

## 3. proc r restart; clears all R objects including functions

`proc r restart;` destroys the entire R subprocess and starts a fresh one.
Any function, library, or variable defined before the restart is gone.

**Symptom:** `Error: could not find function "trim_char_widths"` (or any
user-defined function) in the block after `restart`.

**Fix:** redefine helpers at the top of every restart block, or source
them from an external file via `inFile=`:
```sas
proc r restart;
  submit;
    trim_char_widths <- function(df, ...) { ... }  # redefine here
    library(nlme)
    ...
  endsubmit;
run;
```

If you only need a clean data environment (not a clean function
environment), use `rm(list=ls())` inside a regular `proc r` instead
of `restart`.

## 4. png() fails with "X11 is not available" on headless servers

SAS Viya compute pods run on headless Linux with no display server.
Base R's `png()` tries to use X11 as a fallback device and fails.

**Symptom:** `Error: X11 is not available`

**Fix:** always specify `type = "cairo"` for off-screen rendering:
```r
png(paste0(sas$workpath, "plot.png"), width = 800, height = 600,
    type = "cairo")
plot(...)
dev.off()
```
Or use `rplot()` which handles rendering internally without needing a display.

## 5. Third-party R packages may not be installed (locked-down environments)

In pharma Viya deployments, R packages beyond base R must be installed
by an administrator. `library(mmrm)` or `library(survminer)` will fail
if the admin has not installed them.

**Symptom:** `Error: there is no package called 'mmrm'`

**Fix:** use `requireNamespace()` to check and provide a fallback:
```r
if (requireNamespace("mmrm", quietly = TRUE)) {
  library(mmrm)
  # preferred path
} else {
  cat("NOTE: mmrm not installed – falling back to nlme::lme\n")
  library(nlme)
  # fallback path
}
```
Ask your SAS administrator to install required packages via
`sas-pyconfig` or manually into the shared R library.

## 6. rpy2 pandas2ri.activate() is deprecated (R Runner / PROC PYTHON)

In rpy2 ≥ 3.5 (installed by SAS sas-pyconfig), `pandas2ri.activate()`
raises a `DeprecationWarning` that SAS treats as an error.

**Symptom:** `DeprecationWarning: The activate and deactivate are deprecated.`

**Fix:** use `localconverter` context manager instead:
```python
from rpy2.robjects        import pandas2ri
from rpy2.robjects.conversion import localconverter

with localconverter(pandas2ri.converter):
    r_df = ro.conversion.py2rpy(pandas_df)   # pandas → R

with localconverter(pandas2ri.converter):
    pandas_df = ro.conversion.rpy2py(r_df)   # R → pandas
```

## Quick Reference Card

| Pitfall | Root cause | One-line fix |
|---------|-----------|--------------|
| Column not found after `sd2df()` | Case mismatch | `names(df) <- toupper(names(df))` |
| `$32767` character columns | Parquet engine loses length metadata | `symput("lens", sas_length_stmt(df))` then `data w.ds; &lens. set w.raw;` |
| Function not found after `restart` | `restart` clears entire R session | Redefine helpers inside restart block |
| `X11 is not available` | Headless server, no display | `png(..., type="cairo")` or use `rplot()` |
| Package not found | Not installed on Viya server | `requireNamespace()` + fallback + ask admin |
| `pandas2ri.activate()` deprecated | rpy2 ≥ 3.5 API change | Use `localconverter(pandas2ri.converter)` |
